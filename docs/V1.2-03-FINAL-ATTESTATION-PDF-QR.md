# V1.2-03 - Attestation finale officielle

## Strategie retenue
- Une attestation finale canonique est emise par decision finale (`credit_decisions`) et rattachee au dossier credit via la table existante `attestations`.
- Le depot visible ne contient pas la table `generated_documents` mentionnee dans la roadmap; cette mission reutilise donc les structures documentaires reelles du repo: `attestations`, `verify_tokens`, `documents` quand une parcelle est liee, et le stockage via `storagePut`.
- Le token public de verification est stocke sous forme de hash dans `verify_tokens`, tandis qu'un `verifyCode` public est conserve dans `attestations` pour alimenter le PDF et le lien `/verify?token=...`.

## Conditions d'emission
- Autorisee uniquement pour les dossiers `APPROVED` ou `REJECTED`.
- Une decision finale persistante est obligatoire.
- L'emission est idempotente a l'echelle d'une decision: si une attestation existe deja pour cette decision, elle est retournee telle quelle.

## PDF / verification
- Le PDF est genere cote serveur sans dependance externe lourde, sous forme de document PDF minimal institutionnel.
- Faute de librairie QR dediee dans le depot, le PDF inclut un marqueur visuel de verification et surtout un lien public fonctionnel vers `/verify?token=...`.
- `/verify` expose seulement les metadonnees minimales du document final: categorie, reference, decision synthetique et date d'emission.

## Points d'extension
- Remplacer le marqueur visuel par un QR standard si une librairie QR/PDF est introduite plus tard.
- Ajouter une politique de revocation/versionnement plus fine des attestations.
- Raccorder un futur archivage documentaire ou une page admin documents dediee si le module admin est etendu.
